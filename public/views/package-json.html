¿Qué es el package json?

En este capítulo, hablaremos sobre el package.json que es una pieza muy importante de nuestro proyecto, es tan importante que si ese archivo no existe no hay proyecto , así de importante es esta pieza.

Para generar un package.json tenemos varias opciones.

  • Una sería crearlo nosotros directamente
  • La segunda es con una herramienta que nos ayuda a crearlo

Así que ocuparemos esa herramienta, escribimos dentro de la consola npm init.

Estas líneas quieren decir inicia un node package manager.

Nos saldrán las siguientes preguntas:

Aquí es la importancia de escribir el nombre de la carpeta del proyecto sin mayúsculas y espacios, ya que si package name tiene mayúsculas, dígitos o espacios en blancos se considera un nombre de proyecto incorrecto.

Por lo tanto el nombre del proyecto debe ir siempre sin mayúsculas, dígitos y espacios en blancos.

Esto se utiliza principalmente en proyectos que van escalando poco a poco y que se van actualizando para saber que tipo de versión es y cuanto cambio habido en este proyecto.

El número de versión del programa suele ir definido por algo que se denomina semantic versioning

El sistema de versionado que se sigue en la mayoria de proyectos es el siguiente: MAJOR.MINOR.PATCH

En la versión tenemos (1.0.0) = (MAJOR.MINOR.PATCH)

  • El utimo valor ( PATCH ) en este caso el 0: Es simplemente una correción muy pequeñita de algo que no funcionaba.
  • El segundo valor ( MINOR ) en este caso es 0: Es un cambio menor, que si tenemos otra versión no debería causar conflictos.
  • El primer valor (MAJOR ) en este caso 1: Lo que hace es un cambio gordo en la aplicación, eso implica que si estamos con la versión 1 y actualizan a la versión 2 es posible que sea incompatible y que nuestro proyecto se rompa.

Esto es algo que debemos tener en cuenta en un futuro si escribimos una libreria y la subimos a node y ese tipo de cosas, pero viene bien que nos vaya sonando el nombre de semantic versioning que viene siendo un versionado semántico.

Si hemos trabajado con webpack o con algúnframework el nombre nos sonará y si es la primera vez que tocamos algo de esto, el entry point es el punto de inicio de nuestra aplicación.

Si fuera una página web el entry point sería nuestro index.html , es decir de donde comienza todo.

Normalmente en las aplicaciones de node existe un archivo que se llama app.js que es de donde arranca todo, que es donde está el servidor, las dependencias, digamos que el punto de partida, y desde ahí se accede a todo lo demás.

git repository quiere decir, ¿donde? tenemos el repositorio de esa aplicación.

Serian las palabras claves por las que queremos que se coloque nuestro proyecto, si por ejemplo estamos haciendo una libreria que trata los strings pues en keywords pondríamos string o cosas así para que la gente que esté buscando una libreria para tratar cadenas de caracteres pues nos encuentre por esas palabras.

En license , se especifica el tipo de licencia que es, en el caso de que sea gratuita(open source) , que sea de pago, privada, todas esas cosas.

Creación del package.json más común cuando se empieza

Cuando recién empezamos con node es muy común generar el package.json con el siguiente comando.

  • npm init -y
  • npm init --yes
  • Podemos escribirlo de una u otra forma.

Lo que hace este comando es generar el package.json que equivale a que si nosotros hubiéramos hecho enter a cada línea que nos pregunta.

Entonces cuando se empieza esto es lo más común que se suele hacer, porque en los primeros proyectos no nos ponemos a pensar en todas estas cosas, porque lo único que estamos haciendo es probar node y ver como funciona.

Entonces cuando veamos que alguien está escribiendo npm init -y prácticamente lo que está haciendo es generar el package.json de manera automática.
